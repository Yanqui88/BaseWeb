El usuario está construyendo una plataforma web multi-tenant modular enfocada inicialmente en ecommerce, pero diseñada desde el principio para evolucionar hacia un sistema reutilizable capaz de generar    
   múltiples tipos de sitios para distintos clientes (tiendas, landing pages, blogs, paneles administrativos, etc.). El objetivo principal no es crear “una tienda”, sino desarrollar un framework/skeleton     
   propio reutilizable que permita levantar nuevos proyectos rápidamente compartiendo arquitectura, backend, componentes y lógica común.                                                                        
                                                                                                                                                                                                                
   La visión del sistema es:                                                                                                                                                                                    
                                                                                                                                                                                                                
   * Poder crear múltiples webs para distintos clientes.                                                                                                                                                        
   * Reutilizar módulos/componentes entre proyectos.                                                                                                                                                            
   * Tener un único stack consistente.                                                                                                                                                                          
   * Escalar desde 1 cliente por servidor → hasta multi-tenant real compartiendo infraestructura.                                                                                                               
   * Mantener separación lógica entre tenants mediante tenantId y tenantSlug.                                                                                                                                   
   * Eventualmente hostear múltiples clientes en un mismo servidor propio cuando sea rentable.                                                                                                                  
   * Mientras haya pocos clientes, permitir despliegues separados (1 tenant = 1 servidor) para no absorber costos de hosting.                                                                                   
                                                                                                                                                                                                                
   Arquitectura elegida:                                                                                                                                                                                        
                                                                                                                                                                                                                
   * Monorepo con pnpm workspaces.                                                                                                                                                                              
   * apps/api → backend NestJS.                                                                                                                                                                                 
   * apps/admin → panel administrativo Next.js.                                                                                                                                                                 
   * apps/store → frontend público Next.js.                                                                                                                                                                     
   * packages/* → librerías compartidas futuras (UI, config, hooks, tipos, etc.).                                                                                                                               
                                                                                                                                                                                                                
   Tecnologías elegidas:                                                                                                                                                                                        
                                                                                                                                                                                                                
   * TypeScript en todo el stack.                                                                                                                                                                               
   * NestJS para backend.                                                                                                                                                                                       
   * Prisma ORM.                                                                                                                                                                                                
   * PostgreSQL.                                                                                                                                                                                                
   * Next.js para frontend/admin.                                                                                                                                                                               
   * TailwindCSS.                                                                                                                                                                                               
   * Docker para desarrollo y DB.                                                                                                                                                                               
   * pnpm para monorepo/workspaces.                                                                                                                                                                             
                                                                                                                                                                                                                
   Motivos detrás de TypeScript:                                                                                                                                                                                
                                                                                                                                                                                                                
   * Mantener un único lenguaje en frontend y backend.                                                                                                                                                          
   * Tipado compartido.                                                                                                                                                                                         
   * Mejor DX y escalabilidad.                                                                                                                                                                                  
   * Reducir errores grandes al crecer el proyecto.                                                                                                                                                             
   * Facilitar reutilización entre apps.                                                                                                                                                                        
   * Evitar fricción cognitiva de múltiples lenguajes.                                                                                                                                                          
   * Aunque Go fue considerado por performance y concurrencia, se decidió inicialmente mantener TypeScript por simplicidad arquitectónica y velocidad de desarrollo.                                            
                                                                                                                                                                                                                
   Conversaciones importantes sobre Go vs TypeScript:                                                                                                                                                           
                                                                                                                                                                                                                
   * El usuario preguntó si Go sería mejor para manejar concurrencia cuando muchos tenants operen simultáneamente.                                                                                              
   * Se discutió que TypeScript/Nest puede escalar bastante bien para el tamaño esperado inicial.                                                                                                               
   * Se explicó que un sistema puede mezclar lenguajes mediante microservicios/APIs.                                                                                                                            
   * Se habló sobre comunicación entre módulos mediante HTTP/gRPC/colas/puertos.                                                                                                                                
   * Se aclaró que Docker no es obligatorio para comunicación entre servicios.                                                                                                                                  
   * Se concluyó que Go podría usarse más adelante en servicios específicos críticos, pero no como decisión inicial.                                                                                            
                                                                                                                                                                                                                
   Modelo multi-tenant:                                                                                                                                                                                         
                                                                                                                                                                                                                
   * Se eligió schema global compartido con tenantId en tablas.                                                                                                                                                 
   * No se eligió database-per-tenant ni schema-per-tenant.                                                                                                                                                     
   * Razones:                                                                                                                                                                                                   
                                                                                                                                                                                                                
     * Más simple de administrar.                                                                                                                                                                               
     * Más económico.                                                                                                                                                                                           
     * Más fácil de migrar.                                                                                                                                                                                     
     * Más práctico para comenzar.                                                                                                                                                                              
     * Mejor para pocos clientes inicialmente.                                                                                                                                                                  
   * tenantSlug identifica el tenant públicamente.                                                                                                                                                              
   * URLs actuales:                                                                                                                                                                                             
                                                                                                                                                                                                                
     * /public/:tenantSlug/*                                                                                                                                                                                    
     * /admin/:tenantSlug/*                                                                                                                                                                                     
   * Se conversó sobre permitir dominios propios por tenant:                                                                                                                                                    
                                                                                                                                                                                                                
     * remeras.com.ar                                                                                                                                                                                           
     * peluches.com                                                                                                                                                                                             
   * Se discutió mapping de dominios → tenant internamente.
